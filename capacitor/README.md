# OmniDM Mobile (Capacitor â€” iOS + Android)

A thin native mobile shell around the existing OmniDM web client (`../web`).
Per the **hybrid model**, the AI Dungeon Master engine runs **inside the native
WebView** (the same `engine.bundle.js` the browser and Tauri desktop builds use),
so there is **no Node sidecar** and no server bundled into the app.

- **Play on this device**: the engine runs in-app with *your* model and API key,
  stored only on this device (WebView localStorage) and sent only to the LLM
  endpoint you configure.
- **Connect to a server**: point at an OmniDM server (`npm run web` elsewhere)
  for multiplayer over the unchanged WebSocket protocol.

Nothing here changes the Node path â€” `npm run typecheck` / `npm run smoke` and
every chat adapter keep working exactly as before.

## Why native HTTP (the CORS story)

In a normal browser, "Play on this device" calls your LLM endpoint with a plain
cross-origin `fetch`, which the endpoint must permit via CORS. OpenRouter, OpenAI
and Anthropic do; some hosts (and many local/self-hosted gateways) do not â€” and a
mobile WebView is still a browser, so it enforces CORS too.

On device we route the request through **`CapacitorHttp`** instead: the call runs
on the **native** side (URLSession on iOS, OkHttp on Android), which is not a
browser context and has **no CORS check** and is **not gated by the page CSP's
`connect-src`**. Any LLM host becomes reachable.

The selection is automatic and feature-detected â€” see
[`../src/browser/native-http.ts`](../src/browser/native-http.ts):

- `selectFetch()` returns a `CapacitorHttp`-backed `fetch` **only** when
  `window.Capacitor.isNativePlatform()` is true and the plugin is registered;
- otherwise it returns `undefined`, and the providers keep their default fetch â€”
  so a plain browser and the Node server are completely unaffected.

The native fetch is handed to the providers via `buildProvider({ â€¦, fetchImpl })`
(both the OpenAI SDK and the native Anthropic provider accept it). The plugin is
enabled in [`../capacitor.config.ts`](../capacitor.config.ts).

## Prerequisites (to actually build)

Capacitor wraps the web assets in a real native project, so a platform toolchain
is required. This repo's CI box has **neither**.

| Target | Requirements |
|---|---|
| **Android** | Android Studio + **Android SDK** (`ANDROID_HOME` set), a JDK (17+), Gradle (the wrapper is generated). Emulator or a device with USB debugging. |
| **iOS** | **macOS** + **Xcode** + CocoaPods (`sudo gem install cocoapods`). A simulator or a device + an Apple developer signing profile. iOS **cannot** be built off a Mac. |
| **Both** | Node â‰Ą 22 + this repo's `npm install` (provides `@capacitor/cli`). |

## Build â€” Android

```bash
# from the repo root
npm install                     # brings in @capacitor/{core,cli,android,ios}
npm run build:web               # refresh web/engine.bundle.js if the engine changed

npx cap add android             # one-time: generate android/ from the template
npm run cap:sync                # copy web/ into the native project + update plugins
npm run cap:android             # build & launch on an emulator/device (cap run android)
#   â€¦or open the project in Android Studio to build a signed APK/AAB:
npx cap open android
```

`npm run cap:sync` re-copies `web/` (including a freshly built `engine.bundle.js`)
into the native app â€” run it after any change to the client or the shared engine.

## Where the API key lives (security)

The user's LLM API key is stored **only on the device** (WebView `localStorage`,
v1) and is sent **only** to the endpoint the user configured â€” over the native
`CapacitorHttp` transport on device, or a plain fetch in a browser. It is never
logged, never written into a saved game, and never sent anywhere else.

For a hardened build, move the key into the platform secure store
(iOS Keychain / Android Keystore) via a plugin such as
[`@capacitor/preferences`](https://capacitorjs.com/docs/apis/preferences) with
encryption, or `capacitor-secure-storage-plugin`. localStorage is acceptable for
v1 and is documented here as the current behavior.
